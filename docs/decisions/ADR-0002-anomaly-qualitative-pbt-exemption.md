# ADR-0002: PBT Exemption for Qualitative Anomaly Detection

## عنوان
معافیت از property-based testing برای بخش کیفی تشخیص آنومالی — فقط بخش کیفی، نه سیگنال‌های deterministic.

## وضعیت
Accepted (۲۰۲۶-۰۷-۱۱)

## زمینه (Context)
فیچر 002 (anomaly-flagging) دو نوع سیگنال تشخیص آنومالی دارد:

۱. **deterministic (Category 2):** مثل qty > ۱۰۰، base > ۱۰ میلیون. این‌ها آستانه‌های
   تجاری قابل‌پیکربندی‌اند (env-var)، ولی ذاتاً دترمینیستیک و قابل property testing.
   این‌ها در `app/decision/rules.py` پیاده می‌شوند و property test دارند (T0.1).

۲. **کیفی (LLM-only):** مثل «ترکیب عجیب اقلام»، «لحن مشکوک»، «الگوی غیرعادی نسبت
   به رفتار مشتری». این‌ها ذاتاً قضاوت کیفی‌اند و از LLM می‌آیند — هیچ invariant
   ریاضی ندارند.

قانون constitution می‌گوید: «حداقل پوشش تست ۸۰٪ برای منطق حیاتی؛ برای همین بخش
حیاتی، در صورت امکان از property-based testing استفاده شود.»

## تصمیم
- **بخش deterministic:** subject to PBT (همانطور که در T0.1 پیاده شد — ۳ property test).
- **بخش کیفی (LLM-only):** معاف از PBT. دلیل: «مشکوک بودن» یک قضاوت کیفی است، نه یک
  رابطهٔ عددی مثل `price ≥ 0`. هیچ invariant ریاضی وجود دارد که Hypothesis بتواند
  آن را برای هر ورودی تصادفی بررسی کند. تحمیل PBT در اینجا مصنوعی و بی‌معنی است.

این معافیت **صریح و مستند** است (این ADR) — نه بی‌صدا. سیستم می‌گوید «این قانون
اینجا صدق نمی‌کند» و دلیل می‌آورد، دقیقاً همان رفتاری که آزمایش تست منفی قرار بود
اثبات کند.

## پوشش تست
- بخش deterministic: property test (Hypothesis) + unit test — ۱۰۰٪ coverage.
- بخش orchestration (service.py): unit test با DummyLLM — ≥۸۰٪ coverage (AC7).
- بخش LLM-only: فقط smoke test با DummyLLM (۳ مسیر: clean, flagged, skipped).

## پیامدها (Consequences)
- **مثبت:** آزمایش تست منفی موفق بود — سیستم توانست بین «نیاز به PBT» و «عدم نیاز»
  تفکیک قائل شود و معافیت را صریح مستند کند.
- **منفی:** ریسک اینکه آینده-developerها بخش کیفی را بدون تست رها کنند. کاهش با:
  smoke test با DummyLLM همچنان الزامی است (۳ مسیر).
- **خنثی / ریسک:** اگر روزی یک invariant ریاضی برای بخش کیفی کشف شود (مثلاً
  «هیچ درخواست flagged نباید total > ۱ میلیارد داشته باشد»)، باید property test
  اضافه شود و این ADR بازبینی گردد.
- **پیگیری لازم:** هنگام افزودن سیگنال کیفی جدید، بررسی شود آیا invariant جدیدی
  هم قابل تعریف است یا خیر.

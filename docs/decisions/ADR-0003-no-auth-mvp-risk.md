# ADR-0003: No-Auth in MVP as a Known Risk

## عنوان
endpointهای CRUD کاتالوگ (فیچر 003) در MVP بدون احراز هویت پذیرفته می‌شوند — یک ریسک شناخته‌شده، نه غفلت.

## وضعیت
Accepted (۲۰۲۶-۰۷-۱۱)

## زمینه (Context)
فیچر 003 (catalog-management) پنج endpoint CRUD روی جدول `products` اضافه می‌کند
(POST/GET/PUT/DELETE). این اولین endpoint‌های **نویسنده** در پروژه‌اند — تا پیش
از این، تنها endpoint (`POST /price` از فیچر 001) فقط می‌خواند. دو نیرو در کار بود:

۱. **هم‌راستایی با فیچر 001.** فیچر 001 هم auth را explicit «out of scope» گذاشت
   با فرض اینکه context از سمت فراخواننده می‌آید و endpointها پشت reverse proxy از
   شبکهٔ داخلی (trusted zone) در دسترس‌اند.

۲. **سرعت MVP و عدم وابستگی به زیرساخت اضافی.** افزودن auth (API key، OAuth،
   JWT) یک دامنهٔ کامل است: ذخیرهٔ credential، چرخش کلید، میان‌افزار، تست‌های
   امنیتی جدا. این، scope این فیچر را دو برابر می‌کرد.

گزینهٔ جایگزین مدنظر: یک API key ساده از هدر با env var. رد شد چون (الف) با
فرض شبکهٔ داخلی، ارزش حاشیه‌ای دارد، (ب) نیاز به میان‌افزار و تست امنیتی داشت
که scope را گسترش می‌داد.

## تصمیم
endpointهای CRUD کاتالوگ **بدون auth** پیاده می‌شوند، دقیقاً با همان فرض فیچر 001:
API فقط از شبکهٔ داخلی (trusted zone) قابل‌دسترس است. این تصمیم صریح به‌عنوان
**ریسک شناخته‌شده** مستند می‌شود.

## پیامدها (Consequences)
- **مثبت:** سرعت بالا، scope کمرنگ، هم‌راستایی با فیچر 001، عدم وابستگی به
  زیرساخت credential.
- **منفی:** اگر API به شبکهٔ غیرقابل‌اعتماد برسد، هر کسی می‌تواند کاتالوگ را
  ویرایش/حذف کند.
- **خنثی / ریسک:**
  - هر تغییر قیمت یک audit trail پس‌رویدادی در server log ثبت می‌کند (AC9) —
    هدف ردیابی‌پذیری، نه جلوگیری. این مانع پیشگیرانه نیست.
  - **restore-on-reseed (محدودیت شناخته‌شده):** چون `seed_database()` با
    `SELECT` چک می‌کند و اگر محصول نبود `INSERT` می‌کند، حذف یک محصول seed‌شده
    با DELETE، بعد از restart یا اجرای مجدد seed، **دوباره ساخته می‌شود**. یعنی
    DELETE یک محصول seed‌شده فقط تا next-reseed دوام دارد. این رفتار در T3.1 با
    یک تست صریح ثابت می‌شود. اپراتورها باید از این محدودیت آگاه باشند (در
    `data-model.md` یادداشت می‌شود).
- **پیگیری لازم:** وقتی API به شبکهٔ خارج از trusted-zone برسد (یا چند اپراتور
  متمایز لازم شد)، auth باید برگردد. در آن نقطه یک ADR جدید (supersede این ADR)
  لازم است و ممکن است optimistic locking (نسخهٔ ردیف) هم برای PUT هم‌زمان اضافه
  شود.

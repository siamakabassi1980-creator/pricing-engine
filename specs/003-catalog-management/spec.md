# spec.md — 003-catalog-management

## Why
در فیچر 001 کاتالوگ محصولات یک **seed ثابت** با ۸ محصول بود و صراحتاً خارج از scope
آمده بود (`Out of Scope: مدیریت کاتالوگ محصولات`). این یعنی افزودن/ویرایش/حذف
محصول یا تغییر قیمت، نیازمند تغییر کد و deploy مجدد بود. فیچر 003 این محدودیت را
برمی‌دارد: کاتالوگ از طریق API قابل مدیریت می‌شود، بدون restart.

این فیچر همان سفر مشتری / پارادایم محصولِ constitution فعلی را گسترش می‌دهد
(طبق قاعدهٔ تصمیم پروژه‌ جدید در برابر فیچر جدید در `AGENTS.md`) — پس در همین
پروژه است، نه پروژهٔ جدید.

پیوند با فیچرهای دیگر:
- **005 (inventory-check)** به این فیچر وابسته است (موجودی روی کاتالوگ).
- فیچر 001 (pricing) که الان catalog را فقط می‌خواند، بدون تغییر باقی می‌ماند؛
  CRUD لایه‌ای جدا است و `POST /price` همچنان فقط می‌خواند.

## What
مدیریت کامل کاتالوگ محصولات (CRUD) از طریق REST API، با جدول `products` موجود
(بدون تغییر schema). چهار عملیات:

### ورودی/خروجی‌های پیشنهادی
```
POST   /products          → ایجاد محصول (id, name_fa, unit_price)
GET    /products          → فهرست همهٔ محصولات
GET    /products/{id}     → دریافت یک محصول
PUT    /products/{id}     → ویرایش کامل (name_fa, unit_price)
DELETE /products/{id}     → حذف محصول
```

### مدل موجود (دست‌نخورده)
```python
class Product(Base):
    id: str           # primary key (مثلاً "headphone-x")
    name_fa: str      # نام فارسی
    unit_price: Decimal  # ≥ 0 (Category 1) — Numeric(12, 2)
```

## Acceptance Criteria

1. **پنج عملیات CRUD همگی کار می‌کنند** از طریق endpointهای بالا، روی جدول
   `products` موجود (بدون تغییر schema دیتابیس). seed فعلی دست‌نخورده باقی می‌ماند.

2. **قواعد Category 1 (hard) همیشه حفظ می‌شوند** در لایهٔ مدیریت:
   - `unit_price ≥ 0` همیشه — تلاش برای ایجاد/ویرایش با قیمت منفی باید با خطای
     صریح رد شود (HTTP 422)، نه silently به صفر تبدیل شود یا پذیرفته شود.
   - `name_fa` نباید خالی باشد (یک محصول بدون نام منطقی نیست).
   - `id` نباید خالی باشد.

3. **رفتار خطای استاندارد REST:**
   - ایجاد با `id` تکراری → HTTP 409 Conflict (نه 500).
   - GET/PUT/DELETE روی `id` ناموجود → HTTP 404 (نه 500).
   - ورودی نامعتبر (قیمت منفی، فیلد غایب) → HTTP 422.

4. **عدم ویرایش قیمت توسط LLM:** این فیچر کاملاً دترمینیستیک است — هیچ LLM
   دخالت ندارد (نقض نمی‌کند قانون «Decision هرگز LLM»، چون کاتالوگ در لایهٔ
   Decision نیست). قیمت فقط از ورودی اپراتور می‌آید.

5. **seed همچنان idempotent است:** اجرای `seed_database()` پس از CRUD نباید
   بشکند؛ محصولاتی که قبلاً وجود دارند skip می‌شوند (الگوی موجود حفظ می‌شود).

6. **همهٔ عملیات تست‌پذیرند** با SQLite در حافظه (همان الگوی فعلی). نیازی به
   PostgreSQL واقعی برای تست نیست.

7. **نام مستعار برای محصول (slug):** فیلد `id` یک slug انگلیسی است
   (مثلاً `headphone-x`)؛ `name_fa` فارسی است. (طبق قانون زبان سه‌محوری:
   کد/identifier انگلیسی، محتوای کاربر فارسی.)

8. **مستندسازی قرارداد API** در `specs/003-catalog-management/contracts/` یا
   `data-model.md` (همانند فیچر 001).

8. **مستندسازی قرارداد API** در `specs/003-catalog-management/contracts/` یا
   `data-model.md` (همانند فیچر 001).

9. **ردپای تغییر قیمت (audit log):** هر PUT موفق روی `unit_price` باید مقدار
   قبلی و جدید و timestamp را حداقل در **server log** ثبت کند. هدف: مسیر
   انسانیِ تغییر قیمت هم مثل مسیر LLM (که با دو تست adversarial محافظت
   می‌شود) بدون رد نماند. این فقط logging است، نه جدول audit (خارج از scope).

10. **مستندسازی تصمیم «بدون auth»:** در یک ADR صریح ثبت شود (ADR-0003) که چرا
    در MVP، endpointهای CRUD کاتالوگ بدون احراز هویت پذیرفته شده — یک **ریسک
    شناخته‌شده**، نه غفلت.
- **اعتبارسنجی موجودی** (quantity in stock) → فیچر 005 (inventory-check).
- **تغییر schema جدول products** (افزودن ستون) → اگر لازم شد، ADR جدا.
- **احراز هویت/مجوز اپراتور** → در MVP فرض می‌شود endpointها از شبکهٔ داخلی
  می‌آیند (مثل فیچر 001 که auth را out-of-scope گذاشت). تصمیم «بدون auth» در
  **ADR-0003** به‌عنوان ریسک شناخته‌شده مستند می‌شود.
- **مدیریت customer_tiers** (CRUD روی تخفیف‌ها) → دامنهٔ متفاوت؛ فیچر جدا.
- **بازیابی محصول حذف‌شده (soft delete)** → خارج از scope؛ DELETE سخت است.
- **بروزرسانی بخشی (PATCH)** → فقط PUT (update کامل) در scope.
- **صف/بک‌آلودسازی تغییرات** → خارج از scope.

## پرسش‌های Clarify — بسته‌شده (با پاسخ توسعه‌دهنده)

۱. **مجوز/احراز هویت:** بدون auth پذیرفته شد (همان فرض فیچر 001: شبکهٔ داخلی).
   ولی دو افزوده دارد:
   - یک **ADR-0003** صریح ثبت شود که چرا در MVP بدون auth پذیرفته شده — یک
     **ریسک شناخته‌شده**، نه غفلت. (این AC10 شد.)
   - هر PUT موفق روی `unit_price` باید مقدار قبلی/جدید + timestamp را در
     server log ثبت کند (audit) — تا مسیر انسانیِ تغییر قیمت بدون رد نماند،
     مثل مسیر LLM. (این AC9 شد.)

۲. **DELETE:** hard delete (ردیف واقعاً از DB پاک می‌شود). ساده‌تر، با scope
   جور است.

۳. **حذف محصول استفاده‌شده در سفارش:** مشکلی نیست — سفارش‌ها persist
   نمی‌شوند (pricing stateless است)، پس هیچ FK constraint ای نقض نمی‌شود.

۴. **تغییر id در PUT:** id غیرقابل‌تغییر (immutable). فقط `name_fa` و
   `unit_price` قابل ویرایش. (مثلاً PUT `/products/headphone-x` فقط آن ردیف
   را با id ثابت به‌روز می‌کند؛ ارسال id متفاوت در body = خطای 422.)

۵. **معنای PUT:** replacement کامل — هر سه فیلد (`name_fa`، `unit_price`، و
   `id` در body که باید با path id مطابقت کند) لازم‌اند. فیلد غایب = خطای
   422. PATCH خارج از scope است.

۶. **حداکثر طول فیلدها:** همان مقادیر فعلی کافی (`id` varchar(64)، `name_fa`
   varchar(128)).

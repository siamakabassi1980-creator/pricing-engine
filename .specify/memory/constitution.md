# قانون اساسی پروژه — موتور قیمت‌گذاری پویا (Pricing Engine)

## معماری و استک فنی
- پارادایم: **modular monolith سه‌لایه** با جداسازی سخت‌گیرانهٔ نقش‌ها.
- سه لایه‌ی مجزا (هر کدام یک ماژول فیزیکی جدا، نه توابع پشت‌سرهم):
  - **Perception**: parse درخواست آزاد زبان طبیعی به ساختار ماشین‌خوان
    (`PurchaseRequest`). این لایه صرفاً LLM است.
  - **Decision**: اعمال قواعد قیمت/مالیات/تخفیف روی `PurchaseRequest`. این لایه
    کاملاً دترمینیستیک است و **هرگز نباید به LLM اعتماد کند**.
  - **Generation**: تولید پیش‌فاکتور قابل‌فهم + توضیح تخفیف به زبان فارسی.
    این لایه صرفاً LLM است.
- قواعد به دو دسته تقسیم می‌شوند:
  - **Category 1 (hard-coded):** غیرقابل تغییر توسط مدل (مالیات، non-negative
    price، discount ≤ base). همیشه باید حفظ شوند، فارغ از خروجی LLM.
  - **Category 2 (tunable):** قابل تنظیم از تنظیمات/دیتابیس (تخفیف فصلی).
- LLM از طریق **adapter** قابل‌تعویض است؛ یک `dummy-LLM` قابل‌تزریق برای تست
  بدون کلید API واقعی. (توجیه این تصمیم در ADR ثبت می‌شود.)
- استک: FastAPI (Python 3.11+) + SQLAlchemy + PostgreSQL (برای production)،
  SQLite برای تست.
- تست‌ها: Pytest + **Hypothesis** (property-based testing برای invariantهای مالی).
- **MVP شامل:** endpoint قیمت‌گذاری (POST)، سه لایهٔ مجزا، adapter، property-based
  tests روی Category 1.
- **MVP خارج از scope:** رابط کاربری، مدیریت کاتالوگ (seed ثابت)، چندارزی،
  موجودی انبار، پرداخت واقعی.

## حاکمیت کیفیت (ثابت — از shared-kit)
- حداکثر ۳۰۰ خط برای هر فایل؛ عبور از این حد نیازمند ثبت دلیل در ADR است.
- صفر اخطار Linter قبل از تیک‌خوردن هر تسک، و اجرای Linter حداقل یک‌بار در پایان
  هر فاز روی **کل پروژه** (نه فقط فایل‌های همان تسک).
- Type checking واقعی (`mypy --strict` یا `pyright`)، نه فقط ruff.
- ممنوعیت کد مرده (importهای بلااستفاده، توابع فراخوانی‌نشده).
- حداقل پوشش تست ۸۰٪ برای منطق حیاتی (لایهٔ Decision)؛ برای همین بخش، از
  property-based testing (Hypothesis) استفاده شود.
- اگر معیار پذیرشی به زیرساختی وابسته باشد که در محیط توسعه در دسترس نیست
  (مثلاً سرویس LLM واقعی بدون کلید)، بی‌صدا نادیده گرفته نشود: دلیل در
  `status.md` ثبت می‌شود و فیچر «done-with-caveat» علامت می‌خورد، نه «done» کامل.
- بررسی دوره‌ای آسیب‌پذیری وابستگی‌ها (`pip-audit`).
- هر migration باید مسیر rollback معتبر و تست‌شده هم داشته باشد.

## توقف هوشمند (Fail Fast)
اگر یک تسک با یک رویکرد مشخص دو بار پشت‌سرهم شکست خورد، اجرا متوقف می‌شود.
تکرار کورکورانه ممنوع است. قانون «۲ بار» فقط برای حالت «پیش‌نیاز حاضر +
مشخص نیست شکست قطعی است یا تصادفی» طراحی شده. اگر پیش‌نیاز غایب است، یا خطا
از همان تلاش اول قطعی است (مثل ناسازگاری نسخهٔ کتابخانه)، بدون انتظار توقف کن.

## چرخهٔ کامل فقط برای فیچر جدید/معماری‌محور است
برای تغییرات جزئی مستقیم کد بزن.

## امنیت
هیچ رمز، کلید API، یا مقدار محرمانه هرگز داخل کد یا commit قرار نمی‌گیرد؛
همه از طریق متغیرهای محیطی (.env) خوانده می‌شوند.

## بهداشت Git
تمام commitها از قالب Conventional Commits پیروی می‌کنند، با scope برابر
نام ماژول/کامپوننت: `<type>(<scope>): <description>`. شمارهٔ فیچر در فوتر:
`Refs: NNN-name`. همان quality gateها باید در پایپ‌لاین CI هم اجرا شوند.

**تایمینگ تیک‌زدن تسک:** پیش از commit هر تسک، تیک‌زدن چک‌باکس مربوطه در
`tasks.md` باید در **همان commit** باشد — این timing بخشی از تعریف doneِ تسک
است. یک تسک که کدش commit شده ولی چک‌باکسش `[ ]` مانده، در واقع done نیست.
(کشف‌شده پس از سه بار تکرار چک‌باکس‌های تیک‌نخورده در فیچر 001، 002، 003.)

## زبان (سه محور، نه دو)
- کد (نام متغیر، توابع، کامنت فنی): **انگلیسی**.
- مستندات (spec, plan, ADR, وضعیت): **فارسی**.
- **محتوای کاربر-نهایی** (متن پیش‌فاکتور، توضیحات تخفیف): **فارسی**.

## تصمیم‌گیری معماری
هر تصمیم معماری مهم باید به‌صورت ADR در `docs/decisions/` مستند شود. وابستگی
بین فیچرها باید در `ROADMAP.md` یا `PROJECT_STATUS.md` ثبت شود.

## مدیریت Spec Drift
هر تغییر مستقیم کد (حتی جزئی) باید یک خط یادداشت در `status.md` فیچر مربوطه
اضافه کند — تا `spec.md` به‌مرور از کد واقعی عقب نیفتد.
